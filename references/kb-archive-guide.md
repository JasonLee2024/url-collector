# KB Archive Guide — 知识库归档参考

> url-collector v1.1.0 知识库归档模式的操作规范。

## 一、Johnny Decimal 编号体系

### 三层结构

```
区域 (Area)   XX-XX    10 个（00-09, 10-19, ..., 90-99）
  类别 (Category)  XX    每个区域最多 10 个（X0, X1, ..., X9）
    ID (Item)       XX.XX  每个类别最多 100 个（实际 < 10）
```

### 编号分配规则

**匹配现有类别**：最高 ID + 0.01，pad 为两位小数
```
类别 41_Prompt-Engineering/
  ├── 41.01_提示词工程基础范式.md
  ├── 41.02_高级提示技术.md
  └── 41.03 ← 新文件 = 41.03（41.02 + 0.01）
```

**新建类别**：区域范围内取下一个可用的整数
```
40-49_AI-Engineering/  中有 41, 42, 44
→ 新类别 = 43 或 45（需用户确认）
```

**新建区域**：取下一个可用的十位段，需用户确认
```
现有: 00-09, 10-19, 20-29, 30-39, 40-49
→ 新区域 = 50-59（仅当用户确认后创建）
```

### 扫描算法（脚本辅助）

```bash
# 列出 KB 中所有区域
ls -d <KB>/*-*/ 2>/dev/null

# 列出某区域下所有类别
ls -d <KB>/40-49_*/[0-9]*_*/ 2>/dev/null

# 计算某类别下一个 ID
ls <KB>/<区域>/<类别>/*.md 2>/dev/null | sed 's/.*\/\([0-9]*\.[0-9]*\)_.*/\1/' | sort -n | tail -1
# 结果如 41.02 → 下一个 = 41.03
```

## 二、5C 索引更新协议

每次在 KB 中新增文件后，按顺序更新：

| 序号 | 目标 | 操作 |
|:---:|------|------|
| 1 | 类别 README.md | 在索引表中追加一行：`\| 41.03 \| 标题 \| 内容类型 \| 收藏日期 \|` |
| 2 | 区域 README.md | 如新增类别，更新区域的「本区域包含」表 |
| 3 | SITEMAP.md | ① 目录树中追加条目（`├── 41.03_xxx.md <- 🆕 YYYY-MM-DD`）② 最近更新表中追加一行 |
| 4 | TODO.md | 如涉及预留文章，标记为已完成 |
| 5 | by-topic/ | 如涉及跨域演化主线，追加交叉引用 |

### SITEMAP.md 更新格式

**目录树追加**（保持在正确的层级位置）：

```
├── 40-49_AI-Engineering/
│   └── 41_Prompt-Engineering/
│       ├── 41.01_提示词工程基础范式.md
│       ├── 41.02_高级提示技术.md
│       └── 41.03_新文件名.md  <- 🆕 2026-05-18
```

**最近更新表追加**：

```markdown
| 2026-05-18 | 41.03_新文件名.md | 新收藏：网页资源记录 |
```

## 三、KB 兼容 frontmatter 格式

KB 使用 Markdown 表格 frontmatter（非 YAML），目标 KB 格式优先。

### 通用 KB 格式模板

```markdown
# {标题}

| 属性 | 值 |
|------|-----|
| 编号 | 41.03 |
| 类型 | 网页资源记录 |
| 来源 | {URL} |
| 收藏日期 | YYYY-MM-DD |
| 语言 | zh / en |
| 难度 | ⭐⭐ |
| 状态 | 🌱 初稿 |
| 采集方式 | url-collector |
```

### 与 resource-metadata 模板的关系

- **快速收藏模式**（独立目录）：使用 resource-metadata 通用网页模板（`| 维度 | 字段（中）| ... |`）
- **KB 归档模式**（知识库内）：使用目标 KB 的表格 frontmatter + resource-metadata 元数据表兼容格式

## 四、CLAUDE.md / SITEMAP.md / TODO.md 读写规范

### CLAUDE.md（只读）

KB 根目录的 CLAUDE.md 包含：
- 区域速查表（区域编号 → 名称 → 用途）
- 命名规范
- 文件模板

```markdown
## 区域速查

| 区域 | 名称 | 用途 |
|------|------|------|
| 00-09 | 术语库与平台总览 | 术语、设计哲学 |
| 10-19 | 基础概念 | 核心理论 |
| 20-29 | 工具与框架 | 开发工具 |
| ... | ... | ... |
```

### SITEMAP.md（读写）

两种常见格式：

**格式 A（树形）**：`AI_Basic/SITEMAP.md` — 完整目录树 + 最近更新表

从树形 SITEMAP 可提取的信息：

| 可提取 | 方法 | 用途 |
|--------|------|------|
| 区域列表 | 解析 `├── XX-XX_Name/` 行 | 了解 KB 覆盖的主题域 |
| 类别列表 | 解析 `│   └── XX_Name/` 行 | 了解每个区域下的细分方向 |
| 已有文章 | 解析 `│       ├── XX.XX_title.md` 行 | 推断该类别的主题边界 |
| 新增标记 | 解析 `<- 🆕 YYYY-MM-DD` | 了解活跃更新方向 |

**格式 B（表格）**：`Apple/SITEMAP.md` — 按区域表格 + 文件列表

从表格 SITEMAP 可提取：

| 可提取 | 方法 | 用途 |
|--------|------|------|
| 区域名称 | 解析区域标题行 | 了解 KB 主题域 |
| 文章列表 | 解析表格中的文件名列 | 推断类别边界 |
| 文章描述 | 解析表格中的描述列 | 主题匹配的关键输入 |

**Agent 解析 SITEMAP 的标准步骤：**

1. 读取整个 SITEMAP.md 文件
2. 识别格式类型（树形用 `├──`/`└──` 特征，表格用 `|` 分隔特征）
3. 提取所有类别名称 + 该类别下的文件标题列表
4. 构建：`{类别名: [文件标题1, 文件标题2, ...], ...}` 映射表
5. 从文件标题中提取主题关键词（分词 + 去停用词）

### TODO.md（读写）

```markdown
## 待定文章

| 编号 | 标题 | 优先级 | 备注 |
|------|------|:---:|------|
| 41.03 | （预留） | 中 | |
```

如新文件匹配预留槽位 → 使用预留编号 → 标记已完成。

## 五、Agent 分析协议

> 本节定义 Agent（CC-HAHA / Cursor / Claude Code）在知识库归档模式下，如何系统性地分析 KB 内容结构并确定 URL 的最终目录位置。

### 协议概览

```
Agent 收到: URL + 目标 KB 路径
  ↓
Step 1: 读取 KB 元信息（CLAUDE.md）
Step 2: 构建 KB 内容图谱（SITEMAP.md + 类别 README）
Step 3: 分析 URL 内容（标题/OG/正文 → 关键词向量）
Step 4: 语义匹配 + 候选排序（关键词 vs 类别主题）
Step 5: 推荐 + 用户确认（Top-3 + 匹配理由 + JD 编号）
```

### Step 1：读取 KB 元信息

**目标**：理解 KB 的整体架构和主题覆盖范围。

**操作**：

1. 读取 `<KB>/CLAUDE.md`
2. 提取区域速查表（如有）：`| XX-XX | 区域名称 | 用途描述 |`
3. 构建区域内存模型：

```
AreaMap = {
  "10-19_基础概念": {
    "编号": "10-19",
    "名称": "基础概念",
    "用途": "核心理论、领域定义、发展历史",
    "涵盖主题": ["定义", "历史", "原理", "基础"],
  },
  "40-49_AI-Engineering": {
    "编号": "40-49",
    "名称": "AI工程",
    "用途": "提示词工程、模型部署、工具链",
    "涵盖主题": ["prompt", "部署", "推理", "优化", "工具"],
  },
  ...
}
```

4. 如 CLAUDE.md 不存在 → 降级为仅靠目录名推断

### Step 2：构建 KB 内容图谱

**目标**：了解 KB 中每个类别已有哪些内容，建立「类别→主题」映射。

**操作**：

1. 读取 `<KB>/SITEMAP.md`
2. 按 SITEMAP 解析策略（第四节）提取所有类别及其文件标题
3. 构建类别内存模型：

```
CategoryMap = {
  "41_Prompt-Engineering": {
    "编号": "41",
    "所属区域": "40-49_AI-Engineering",
    "已有文章": [
      "41.01_提示词工程基础范式",
      "41.02_高级提示技术-CoT-ToT-ReAct",
      "41.03_纳瓦尔神级提示词",
    ],
    "主题关键词": ["提示词", "prompt", "CoT", "ToT", "ReAct", "基础", "高级"],
  },
  ...
}
```

4. **可选增强**：读取该类别下1-2篇代表性文章的标题/首段，抽取更精准的主题向量
5. 统计信息：每个类别的文件数、最新更新日期

### Step 3：分析 URL 内容

**目标**：从 URL 页面提取结构化的内容画像。

**操作**：

1. 运行 `collect.sh <URL>` 或 WebFetch 提取页面信息
2. 提取以下内容：
   - **标题**（`<title>` / OG title）
   - **摘要**（OG description / meta description）
   - **站点名**（OG site_name）
   - **正文首段**（如有全文提取）
   - **标签**（OG tags / meta keywords）
3. 生成内容摘要（2-3 句话，概括页面核心内容）
4. 提取主题关键词（5-10 个，中英文混合，去除通用停用词）：

```
URL 关键词示例（agentskill.work）:
["Claude Skill", "技能发现", "GitHub", "项目聚合", "搜索", "skills.sh", "开源"]
```

### Step 4：语义匹配 + 候选排序

**目标**：计算 URL 内容与每个 KB 类别的匹配度，产出排序列表。

**匹配策略（三级 + 权重修正）：**

| 匹配度 | 条件 | 示例 |
|--------|------|------|
| **高 (HIGH)** | URL 关键词与类别「主题关键词」有 ≥2 个命中 | URL 含 "prompt, CoT, ReAct" vs 类别 "提示词工程" → 高 |
| **中 (MEDIUM)** | URL 关键词与类别有 1 个命中，或与区域描述高度相关 | URL 含 "AI" vs 区域 "AI-Engineering" → 中 |
| **低 (LOW)** | 无关键词命中，仅通过站点名/内容类型勉强关联 | URL 是 "GitHub README" vs 类别 "文学理论" → 低 |

**权重修正因子：**

匹配分数计算后，应用以下修正：

| 因子 | 规则 | 修正幅度 |
|------|------|:---:|
| **饱和度惩罚** | 类别已有 ≥8 篇文章 → 降低优先级（接近满 10 篇） | -0.5 |
| **活跃度奖励** | 类别在最近 30 天内有新增/更新 → 提升优先级 | +0.3 |
| **新鲜度惩罚** | 类别 90 天内无更新 → 降低优先级（可能过时） | -0.3 |

**"不应归档"判定（LOW-GATE）：**

如果 URL 内容与 KB **所有**类别的匹配度均为 LOW，Agent **必须**明确提示：

```
⚠️ 未找到高度相关的类别。该 URL 内容与当前 KB 的主题覆盖范围可能存在较大偏差。
建议：① 不归档到此 KB；② 寻找更相关的 KB；③ 如仍要归档，由用户手动指定位置。
```

此判定防止将无关内容强行入库。

**操作**：

1. 遍历 CategoryMap，计算每个类别的匹配分数
2. 施加权重修正（饱和度/活跃度/新鲜度）
3. 输出 Top-3 候选：

```
Top-3 候选（url-collector: agentskill.work → AI_Basic）:

1. 41_Prompt-Engineering [HIGH] ← 修正后
   - 命中关键词: "Claude Skill"(→"技能"), "skill"(→"prompt工程技能")
   - 已有文章: 41.01~41.03（提示词技术）
   - 推荐编号: 41.04
   - 饱和度: 3/10 ✅ | 最近更新: 2026-05-15 (+0.3)

2. 00-09_术语库与平台总览 / 02_工具与平台 [MEDIUM]
   - 命中: "平台"(→"GitHub项目聚合平台")
   - 已有文章: 02.01~02.05（工具平台介绍）
   - 推荐编号: 02.06
   - 饱和度: 5/10 ✅ | 最近更新: 2026-04-10 (-0.3)

3. 90-99_社区与生态 [MEDIUM]
   - 命中: "开源"(→"开源生态")
   - 已有文章: 90.01~90.03
   - 推荐编号: 90.04
   - 饱和度: 3/10 ✅ | 最近更新: 2026-01-20 (-0.3)
```

### Step 5：推荐 + 用户确认

**目标**：呈现分析结果，获得用户决策。

**输出格式**：

```
>>> 分析结果：将 https://agentskill.work/zh 归档到 AI_Basic 知识库

页面摘要：Claude Skill 项目聚合平台，自动汇总 GitHub 热门 Skill 项目（1030+），支持搜索、分类浏览

推荐归属：

  1. [HIGH] 41_Prompt-Engineering — 主题高度相关（技能/Claude Skill），分配编号 41.04
  2. [MEDIUM] 02_工具与平台 — 平台属性匹配，分配编号 02.06
  3. [MEDIUM] 90_社区与生态 — 开源生态相关，分配编号 90.04

  是否选择推荐 #1？或者指定其他位置？[1/2/3/手动指定]
```

**用户选择后的操作**：

- 选择 1/2/3 → 使用对应编号，执行 `collect.sh --kb <KB> --area <Area> --category <Cat>`
- 选择手动指定 → 用户提供 area/category → 验证存在 → 执行
- 选择新建类别 → 评估区域内的下一个可用整数 → 创建目录 → 执行

### 协议总结（Agent 检查清单）

在 KB 归档模式下，Agent 必须完成以下检查：

- [ ] 已读取目标 KB 的 CLAUDE.md（区域速查表）
- [ ] 已读取目标 KB 的 SITEMAP.md（内容图谱）
- [ ] 已构建 AreaMap + CategoryMap
- [ ] 已提取 URL 的主题关键词（5-10 个）
- [ ] 已生成 Top-3 候选 + 匹配理由
- [ ] 已呈现给用户确认
- [ ] 已确认的编号不冲突
